Documentation

This page tells you how Howl operates on the inside. Read it before you repair a defect, before you add a feature, or when you want to examine the source of a real DAW. This page gives the large picture. The reference documents at the bottom give more data.

The software base

Howl is C++20 code on the JUCE 8 framework. The build uses CMake and Ninja. The tests use Catch2. The project has four third party dependencies after JUCE: Catch2 for the tests, the CLAP headers and the clap helpers for the CLAP support, and Rubber Band for the time stretch function. CMake gets all of them automatically, at fixed versions. Thus a new clone builds with no manual steps.

How to build and run

Install CMake 3.22 or newer, Ninja, and a C++20 compiler. Then do these commands:

cmake -B build -G Ninja
cmake --build build

The app binary goes to build/src/app/Howl_artefacts/. To run the tests, do this command:

ctest --test-dir build

The module layout

The source tree has modules with one direction of dependency. A lower module does not know the modules above it. ui can see model, but model does not know ui.

ModuleWhat lives there
src/coreThe basic shared types: the sample count and the audio block type that each process call uses. This module has no dependencies.
src/dspThe sound code: the subtractive synth, the sampler, the included effects (EQ, compressor, limiter, delay, reverb, gain), and the offline time stretcher. This module does not use JUCE. Thus it is portable, and you can test it alone.
src/engineThe abstract parts that the audio thread runs: the transport (position, tempo, loop), the process graph and the nodes, and the Effect and Instrument interfaces. Each part that makes sound implements one of these interfaces.
src/ioThe connection to the machine: the audio device, the WAV file read and write code, and the MIDI input hub. The hub puts the hardware events in a queue for the engine.
src/modelThe project data: the tracks, the clips, the notes, the patterns, the session grid, the mixer with the buses and the sends, the automation, and the undo stack with each undo command. This module also has the arrangement node. There the model meets the audio thread.
src/pluginsThe plugin support: one shared instance interface with the VST3 and CLAP adapters behind it, the scanner, and the sandboxed instance. The sandboxed instance runs a plugin in a separate process.
src/hostThe small separate binary that holds a sandboxed plugin. If a plugin has a crash, this process dies. Howl continues.
src/projectThe serialization code. It changes the full model into JSON and back.
src/uiAll views: the arrange timeline, the piano roll, the session grid, the channel rack, the mixer, the browser, and the automation editor, with the one theme that all views share.
src/appMain.cpp owns and connects all parts. It is the only place that can know all the modules above.

The one rule

Code that runs on the audio thread must obey this rule. This is all code that a process() call can touch. The code must not allocate memory. It must not take a lock. It must not wait on a system call. It must not throw an exception. It must not write a log. The audio thread has a hard deadline of some milliseconds. If the code misses the deadline, you hear a defect in the output. This rule breaks a DAW when the code does not obey it. Much of the design in Howl exists for this rule: buffers with early allocation, lock free queues for messages into the audio thread, and atomic reads for values that the UI changes.

Before you write audio code, know the thread that your code runs on. This is the most important habit.

The render of one audio block

The audio device calls Howl one time for each block. A block is some hundreds of samples. Each callback moves the transport forward. Then it tells the process graph to fill the block. The main node of the graph is the arrangement node. This node goes through each track. It makes the MIDI clips into sound through the instrument of each track. It plays the audio clips. It reads the automation. It gives the buffer of each track to the mixer. The mixer applies the effects, the gain, and the pan of each strip. It sends the tracks into the buses and the sends. It sums all audio through the master strip into the block for the device. The session clips, the pattern playback, the frozen tracks, and the sample previews all occur in this same pass.

Communication between the UI and the engine

The UI does not touch the audio directly. Edits go through command objects: add a note, move a clip, remove a track. One undo stack performs the commands. Thus one mechanism makes each change reversible. The same counter controls the autosave and the guard dialogs for changes that are not saved. Views hold references into the model. A timer makes the views paint again from the model. Data that must go to the audio thread crosses through the lock free queues and the atomic values above. Main.cpp connects the callbacks between the views and the app at the start. Thus this file is the map of all connections.

Plugins and the sandbox

The VST3 and CLAP plugins are behind one shared interface. Thus the rest of the app does not know the format of a plugin. The default for each plugin is a sandbox. A separate host process loads the plugin. Audio crosses through shared memory with a wait limit. Control messages move as JSON lines. If the plugin has a crash, only the child process dies. Howl finds the dead child, sets the plugin to bypass, and shows this in the UI. One click starts the plugin again.

Project storage

A project saves as plain JSON text in a .howl file. The file holds the tracks, the clips, the notes, the mixer state, the patterns, the automation, the MIDI mappings, and the instrument state. An audio clip keeps the path to the source file, not the audio data. While you work, a timer saves a changed project to an adjacent .autosave file. When you open a project with a newer autosave, the app asks you if it must recover the autosave.

Tests

Approximately two hundred Catch2 tests examine the engine, the mixer, the DSP, the plugin adapters, the serialization, and the sandbox protocol. The strongest tests are the golden render checks. These tests make a known project into audio offline. They compare the result with a stored reference, sample by sample, on each platform. If a change makes a different sound for old projects, these tests fail. That is their purpose.
