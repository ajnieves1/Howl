Documentation

This page is for people who want to understand how Howl works on the inside, whether to fix a bug, add a feature, or just read a real DAW's source. It covers the lay of the land; the deeper reference docs are linked at the bottom.

What it's built on

Howl is C++20 on the JUCE 8 framework, built with CMake and Ninja, tested with Catch2. There are four third party dependencies beyond JUCE: Catch2 for tests, the CLAP headers for CLAP hosting, and Rubber Band for time stretching. CMake fetches and pins all of them automatically, so a fresh clone builds with no setup.

Building and running

You need CMake 3.22 or newer, Ninja, and a C++20 compiler.

cmake -B build -G Ninja
cmake --build build

The app binary lands in build/src/app/Howl_artefacts/. Run the tests with:

ctest --test-dir build

Where things live

The source tree is split into modules with one direction of dependency: lower modules never know about higher ones. ui can see model, but model has never heard of ui.

ModuleWhat lives there
src/coreThe smallest shared vocabulary: sample counts, the audio block type that every processing call passes around. Nothing here depends on anything.
src/dspSound itself: the subtractive synth, the sampler, the built in effects (EQ, compressor, limiter, delay, reverb, gain), and the offline time stretcher. Deliberately JUCE free, so it stays portable and testable in isolation.
src/engineThe abstractions the audio thread runs on: the transport (position, tempo, looping), the processing graph and its nodes, and the Effect and Instrument interfaces everything sound making implements.
src/ioTalking to the machine: the audio device, WAV file reading and writing, and the MIDI input hub that queues hardware events for the engine.
src/modelThe project itself: tracks, clips, notes, patterns, the session grid, the mixer with buses and sends, automation, and the undo stack with every undoable command. Also the arrangement node, where the model meets the audio thread.
src/pluginsPlugin hosting: one shared instance interface with VST3 and CLAP adapters behind it, the scanner, and the sandboxed instance that runs a plugin in a separate process.
src/hostThe small separate binary that sandboxed plugins actually run inside. If a plugin crashes, this process dies, not Howl.
src/projectSerialization: turning the whole model into JSON and back.
src/uiEvery view: the arrange timeline, piano roll, session grid, channel rack, mixer, browser, automation editor, and the theme all of them share.
src/appMain.cpp: owns everything, wires everything, and is the only place allowed to know about all of the above at once.

The one rule

Anything that runs on the audio thread, meaning anything reachable from a process() call, may not allocate memory, take a lock, block on a system call, throw, or log. The audio thread has a hard deadline measured in milliseconds; miss it and the output glitches audibly. This is the rule that breaks DAWs when ignored, and most of the less obvious design in Howl exists to honor it: preallocated buffers, lock free queues for messages entering the audio thread, and atomic reads for values the UI edits.

If you contribute one thing to audio code, make it this: know which thread your code runs on before you write it.

How one block of audio happens

The audio device calls Howl back once per block, a few hundred samples at a time. Each callback advances the transport, then asks the processing graph to fill the block. The graph's main node is the arrangement node, which walks every track: it renders MIDI clips through each track's instrument, plays audio clips, evaluates automation, and hands each track's buffer to the mixer. The mixer applies each strip's effects, gain, and pan, routes tracks into buses and sends, and sums everything through the master strip into the block the device asked for. Session clips, pattern playback, frozen tracks, and sample previews all resolve inside that same walk.

How the UI talks to the engine

The UI never touches audio directly. Edits go through command objects (add a note, move a clip, remove a track) performed on a single undo stack, so every change is undoable through one mechanism and the same counter drives autosave and the unsaved changes guards. Views hold references into the model and repaint from it on a timer; anything that must reach the audio thread crosses over through the lock free queues and atomics mentioned above. The wiring between views and the app lives in callbacks that Main.cpp connects at startup, which makes it the map of who talks to whom.

Plugins and the sandbox

VST3 and CLAP plugins sit behind one shared interface, so the rest of the app never cares which format a plugin is. By default each plugin runs sandboxed: a separate host process loads it, audio crosses over through shared memory with a bounded wait, and control messages travel as JSON lines. If the plugin crashes, the child process dies alone; Howl detects it, bypasses the plugin, marks it in the UI, and offers a one click restart.

How projects persist

A project saves as plain JSON text in a .howl file: tracks, clips, notes, mixer state, patterns, automation, MIDI mappings, and instrument state. Audio clips store the path to their source file, not the audio itself. While you work, a timer autosaves dirty projects to a sibling .autosave file, and opening a project with a newer autosave offers to recover it.

Tests

Around two hundred Catch2 tests cover the engine, the mixer, DSP, plugin adapters, serialization, and the sandbox protocol. The strictest of them are the golden render checks: a known project is rendered offline and compared against a stored reference, sample by sample, on every platform. If a change alters the sound of existing projects, these fail, which is exactly the point.
